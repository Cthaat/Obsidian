## 本章速览

- 操作系统是最基本的系统软件：管理资源、组织调度、提供接口和运行环境。
- 四个基本特征是并发、共享、虚拟、异步；并发和共享最基本，二者互为存在条件。
- 发展主线是提高资源利用率和交互/实时能力：手工 -> 单道批处理 -> 多道批处理 -> 分时 -> 实时 -> 网络/分布式/个人机。
- 运行环境重点是用户态/内核态、特权指令、中断/异常、系统调用的状态切换。
- 系统调用“发生/发起”在用户态，服务例程执行在内核态；库函数通常在用户态，是系统调用的上层封装。
- 结构题抓对比：分层清晰但低效，宏内核快但大，微内核稳但切换慢，外核只分配/保护资源。

## 知识点清单

### 1.1 操作系统的基本概念

#### 1.1.1 操作系统的概念

- 计算机系统自下而上可粗分为：硬件、操作系统、应用程序、用户。
- 操作系统 `OS` 是控制和管理整个计算机系统硬件与软件资源，合理组织、调度计算机工作与资源分配，并为用户和其他软件提供接口与环境的程序集合。
- 操作系统是计算机系统中最基本的系统软件。
- OS 的位置：充当用户/应用程序与硬件之间的中介。

#### 1.1.2 操作系统的功能和目标

- 作为资源管理者：
  - 处理机管理：进程/线程管理、进程控制、同步、通信、死锁处理、处理机调度。
  - 存储器管理：内存分配与回收、地址映射、内存保护与共享、内存扩充。
  - 文件管理：文件存储空间、目录、读写、保护。
  - 设备管理：缓冲管理、设备分配、设备处理、虚拟设备，完成用户 I/O 请求。
- 作为用户与硬件之间的接口：
  - 命令接口：用户控制作业。
    - 联机命令接口：交互式，适合分时/实时系统。
    - 脱机命令接口：批处理式，用户提交作业控制说明书，由命令解释程序逐条执行。
  - 程序接口：由系统调用组成，也称广义指令；程序员通过它请求 OS 服务。
  - GUI：严格说不是 OS 的必要组成部分；GUI 最终调用的系统调用属于 OS。
- 作为扩充机器：
  - 裸机：没有软件支持的计算机。
  - OS 把裸机改造成更易用、更强的扩充机器/虚拟机。

#### 1.1.3 操作系统的特征

| 特征 | 含义 | 常考点 |
| --- | --- | --- |
| 并发 | 两个或多个事件在同一时间间隔内发生 | 宏观同时、微观交替 |
| 共享 | 系统资源可供多个并发进程共同使用 | 互斥共享、同时访问 |
| 虚拟 | 把一个物理实体变成多个逻辑对应物 | 时分复用、空分复用 |
| 异步 | 进程以不可预知速度推进 | 走走停停，但相同环境下结果应可再现 |

- 并发 vs 并行：
  - 并发：同一时间间隔内发生，单处理机可通过分时实现。
  - 并行：同一时刻同时发生，需要多处理器、多流水线等硬件支持。
  - 单 CPU 上进程与进程不能真正并行；CPU 与 I/O 设备、I/O 设备与 I/O 设备可并行。
- 共享方式：
  - 互斥共享：一段时间内只允许一个进程访问，如打印机、磁带机、临界资源。
  - 同时访问：宏观上多个进程同时访问，微观上常分时交替，如磁盘、重入代码文件。
- 并发和共享的关系：
  - 没有并发，就没有资源共享问题。
  - 不能有效共享资源，就会影响甚至破坏并发。
- 虚拟技术：
  - 时分复用：如虚拟处理器。
  - 空分复用：如虚拟存储器、虚拟设备。

#### 1.1.4/1.1.5 本节习题精选与答案解析吸收

- OS 管理的是计算机资源，包括硬件资源和软件资源；文件不是硬件，但属于资源。
- 源程序本身不是 OS 关心的管理对象；OS 管理文件的逻辑结构、物理结构和组织方式，不关心文件具体内容。
- 高级语言编译器不是 OS 必须提供的功能。
- 操作系统不是“用来编程的程序”，而是管理资源、扩充裸机、提供接口的程序。
- 系统调用只能通过用户程序间接使用，不是键盘命令。
- 程序员接口 = 系统调用；应用程序接口常考答案也是系统调用，不是库函数、中断、原语。
- 单处理机系统可并行的是处理机与设备、处理机与通道、设备与设备；进程与进程只并发不并行。

### 1.2 操作系统发展历程

#### 1.2.1 手工操作阶段（此阶段无操作系统）

- 用户独占全机，程序装入、运行、输出都靠人工干预。
- 缺点：
  - 资源利用率低。
  - CPU 等待人工操作和 I/O，利用不充分。

#### 1.2.2 批处理阶段（操作系统开始出现）

- 单道批处理系统：
  - 作业脱机输入到磁带，由监督程序自动连续处理。
  - 特点：自动性、顺序性、单道性。
  - 缺点：内存中只有一道作业；作业运行期间等待 I/O 时，CPU 空闲。
- 多道批处理系统：
  - 多个作业在外存后备队列中，作业调度程序选若干作业调入内存。
  - 某道程序等待 I/O 时，CPU 转去执行另一道程序；依赖中断机制。
  - 特点：多道、宏观上并行、微观上串行。
  - 优点：CPU 利用率高，资源利用率高，系统吞吐量大。
  - 缺点：用户响应时间长，不提供人机交互能力。
  - 需解决：处理机分配、内存分配、I/O 分配、程序和数据组织及保护。
- 多道程序设计引入后，程序执行失去封闭性和顺序性，产生竞争、制约和间断性。

#### 1.2.3 分时操作系统

- 分时技术：把处理器时间划分为很短的时间片，多个联机用户轮流使用 CPU。
- 目标：支持交互，快速响应用户。
- 特点：
  - 同时性/多路性：多个终端用户同时使用一台主机。
  - 交互性：用户可通过终端直接控制程序运行。
  - 独立性：用户之间互不干扰。
  - 及时性：用户请求能在较短时间内得到响应。
- 响应时间常用近似：`T = Q x N`，`Q` 为时间片，`N` 为用户数。

#### 1.2.4 实时操作系统

- 实时系统要求在规定时间内处理外部事件，不需要等待时间片轮转。
- 硬实时：必须在规定时刻或时间范围内完成，如飞行控制。
- 软实时：偶尔违反时间要求不会造成永久损害，如订票、银行业务、股票交易。
- 主要目标：及时性和可靠性；资源利用率不是首要目标。

#### 1.2.5 网络操作系统和分布式计算机系统

- 网络操作系统：实现网络中计算机之间的通信、数据传送和资源共享，常见为集中控制式。
- 分布式系统：
  - 多台计算机地位平等，无主从关系。
  - 资源为所有用户共享。
  - 任务可分布到多台计算机并行协同完成。
  - 主要特点：分布性、并行性。
- 本质区别：分布式 OS 强调多计算机协同完成同一任务；网络 OS 更强调网络通信与资源共享。

#### 1.2.6 个人计算机操作系统

- 面向个人计算机，应用广泛，如 Windows、Linux、macOS。
- 其他类型还包括嵌入式 OS、服务器 OS、智能手机 OS。

#### 1.2.7/1.2.8 本节习题精选与答案解析吸收

- 提高单机资源利用率的关键技术：多道程序设计。
- 批处理系统主要缺点：缺少交互能力。
- 操作系统基本类型：批处理、分时、实时。
- 分时系统不是主要用于批处理作业；它面向交互式作业。
- 实时系统的应用判断：航空订票、机床控制、股票交易等需要及时响应外部变化。
- 多道程序技术的前提之一是中断功能。
- 多任务 OS 不一定需要多 CPU；单 CPU 可通过分时实现并发。
- 进程数不是越多 CPU 利用率越高，竞争和死锁可能降低利用率。
- CPU 利用率/甘特图题：
  - 画出 CPU、设备、进程的时间线。
  - 同一资源同一时刻只能给一个进程使用。
  - `CPU 利用率 = CPU 忙碌时间 / 总完成时间`。
  - 单道环境要按作业顺序串行算，多道环境要允许 CPU 与设备并行。

### 1.3 操作系统的运行环境

#### 1.3.1 处理器运行模式

- CPU 执行两类程序：
  - OS 内核程序：管理者，可执行特权指令。
  - 用户程序：被管理者，不能直接执行特权指令。
- CPU 状态：
  - 内核态/管态/核心态：可执行特权指令和非特权指令。
  - 用户态/目态：只能执行非特权指令。
  - 书中示例：模式位 `0` 表示内核态，`1` 表示用户态。
- 特权指令：
  - 不允许用户直接使用。
  - 例：I/O 指令、开/关中断、内存清零、修改 PSW、存取内存保护寄存器、设置时钟、启动/关闭设备、切换进程、设置中断向量、设置中断屏蔽字。
- 非特权指令：
  - 用户可直接执行。
  - 普通数据处理、普通转移/跳转、压栈、寄存器清零、一般存取用户空间数据等。
- 访管指令/陷入指令：
  - 在用户态执行，本身不是特权指令。
  - 作用是主动产生自陷，把 CPU 从用户态转入内核态，请求系统调用。
- OS 内核常见内容：
  - 时钟管理：系统时间、时间片、定时任务、进程运行时间统计。
  - 中断机制：现代 OS 是中断驱动的软件；中断支撑调度、系统调用、设备驱动、文件访问。
  - 原语：低层、接近硬件、短小频繁、原子执行，常通过关中断保证不可分割。
  - 系统控制的数据结构及处理：JCB、PCB、设备控制块、队列、缓冲区、空闲区登记表、内存分配表等。

#### 1.3.2 中断和异常的概念

- 用户态进入内核态的唯一途径：中断或异常。
- 外中断/中断 `Interrupt`：
  - 来自 CPU 执行指令外部的事件。
  - 例：I/O 完成中断、时钟中断、DMA 结束中断。
  - 分为可屏蔽中断 `INTR` 和不可屏蔽中断 `NMI`。
- 异常/内中断 `Exception`：
  - 来自 CPU 执行指令内部的事件，不能被屏蔽。
  - 故障 `Fault`：由当前指令引起，处理后可能回到该指令继续，如缺页、非法操作码、除零、溢出。
  - 自陷 `Trap`：事先安排的异常，如系统调用/访管指令、条件陷阱指令。
  - 终止 `Abort`：无法继续执行的严重硬件故障，如控制器出错、存储器校验错。
- 处理结果：
  - 可恢复：处理后回到第 `i` 条或第 `i+1` 条指令继续。
  - 不可恢复：终止用户程序。
- 中断处理 vs 子程序调用：
  - 中断与当前程序相互独立，随机发生；子程序与主程序有主从关系，由 `CALL` 事先安排。
  - 中断需要硬件电路和向量机制；子程序调用主要是软件过程。
  - 中断入口可由硬件向量法确定；子程序入口由调用指令地址码给出。
  - 二者都保护 PC；中断还必须保护 PSW，子程序调用通常不需要保存 PSW。
  - 多个中断请求需判优；子程序调用无此操作。

#### 1.3.3 系统调用

- 系统调用是 OS 提供给应用程序/程序员使用内核功能的接口。
- 凡是涉及共享资源或系统安全的操作，应通过系统调用请求 OS 代为完成。
- 常见系统调用功能：
  - 设备管理：请求/释放设备、启动设备。
  - 文件管理：读、写、创建、删除文件。
  - 进程控制：创建、撤销、阻塞、唤醒进程。
  - 进程通信：消息传递、信号传递。
  - 内存管理：分配、回收内存，获取作业内存区大小和起始地址。
- 系统调用处理流程：
  - 用户程序传递系统调用号和参数。
  - 执行陷入 `trap` 指令，CPU 从用户态进入内核态。
  - 硬件保存断点和 PSW，并切换模式；OS 保存通用寄存器等现场。
  - OS 查系统调用入口表，转入对应内核服务例程。
  - 服务完成后恢复现场，返回用户态继续执行。
- 用户态转内核态常见事件：
  - 用户程序请求 OS 服务，即系统调用。
  - 发生外部中断。
  - 用户程序产生错误状态，如缺页、除零。
  - 用户程序企图执行特权指令。
- 内核态转用户态：通常由中断返回指令等特权指令实现。
- 库函数 vs 系统调用：
  - 库函数是语言库/应用程序的一部分，通常运行在用户态。
  - 系统调用是 OS 内核接口，服务例程运行在内核态。
  - 部分库函数封装系统调用，部分库函数与系统调用无关。
  - 系统调用有状态切换和上下文保存开销，通常比普通库函数调用开销大。

#### 1.3.4/1.3.5 本节习题精选与答案解析吸收

- “在用户态发生”和“在用户态执行”要分开：
  - 系统调用、外部中断、缺页可在用户态发生。
  - 进程切换不可能在用户态发生，它由 OS 在内核态完成。
- `trap` 指令可以在用户态执行；关中断、开中断、I/O、设置中断屏蔽字必须在内核态执行。
- 系统调用顺序：传递参数 -> 执行 `trap` -> 执行服务程序 -> 返回用户态。
- 系统调用处理中，硬件完成保存断点/PSW 和切换到内核态；OS 完成保存通用寄存器和执行服务例程。
- 中断处理程序一定是 OS 程序，运行在内核态；被中断的程序可能是用户程序，也可能是内核程序。
- 中断或异常“发生时”CPU 可能在用户态或内核态；中断处理程序开始执行时 CPU 一定在内核态。
- 外部中断处理时，PC/PSW 通常由硬件保存，通用寄存器由 OS 保存；TLB/Cache 通常无须保存。
- 缺页属于异常/内中断中的故障，不是外部中断。
- 中断向量表适合用数组，便于按中断号 `O(1)` 定位入口。
- 创建新进程通常通过系统调用完成；页置换和进程调度由 OS 内部完成，不是用户系统调用。

### 1.4 操作系统结构

| 结构 | 核心思想 | 优点 | 缺点/易错 |
| --- | --- | --- | --- |
| 分层法 | 层 0 为硬件，顶层为用户接口，每层只调用紧邻低层 | 结构清晰，便于调试、验证、扩充、维护 | 层次难定义，不够灵活；跨层开销大，效率低 |
| 模块化 | 按功能划分模块和子模块，通过接口通信 | 正确性、可理解性、可维护性好；适应性强；开发快 | 接口难设计；模块间依赖复杂，难确定可靠设计顺序 |
| 宏内核 | 主要功能模块作为紧密整体运行在内核态 | 性能高，模块间调用快 | 内核大，可靠性和可维护性差，模块错可能拖垮内核 |
| 微内核 | 只保留最基本核心功能在内核，其余服务在用户态服务器 | 可靠、安全、易扩展、可移植、支持分布式 | 用户态/内核态切换和消息传递多，性能开销大 |
| 外核 | 内核只分配、回收、保护资源，检查安全性 | 减少资源映射层，用户可自行管理已分配资源 | 外核不负责进程调度等完整 OS 管理 |

- 模块独立性：
  - 内聚性越高越好。
  - 耦合度越低越好。
- 微内核通常放入：
  - 进程/线程间通信机制。
  - 低级进程管理和调度机制。
  - 低级存储器管理机制，如页表和地址变换。
  - 中断和陷入处理。
  - 与硬件紧密相关的低级 I/O。
- 微内核通常不放入：文件系统服务、完整虚拟存储管理策略、复杂高级服务。
- 微内核四个方面：
  - 足够小的内核。
  - 基于客户/服务器模式。
  - 机制与策略分离。
  - 采用面向对象技术。
- 现代主流 OS 多为宏内核或混合内核，不能简单说成纯分层式或纯微内核。

### 1.5 操作系统引导

- 操作系统以数据形式存放在硬盘中，开机后通过引导程序逐级加载。
- 常见引导链条：
  - 激活 CPU，读取 ROM 中的 boot 程序，开始执行 BIOS。
  - BIOS 在内存低地址建立中断向量表，进行 POST 通电自检。
  - BIOS 按 Boot Sequence 选择启动设备。
  - 加载主引导记录 `MBR`。
  - `MBR` 扫描分区表，找到活动分区。
  - 加载分区引导记录 `PBR`。
  - `PBR` 加载启动管理器。
  - 启动管理器加载 OS 初始化程序到内存执行。
- 易记：`ROM/BIOS -> POST -> 启动设备 -> MBR -> 活动分区 -> PBR -> 启动管理器 -> OS`。
- 启动指令细节题：
  - CPU 加电后 `CS:IP` 指向 `FFFF0H` 附近。
  - 先执行跳转到 BIOS 的 `JMP` 指令。
  - BIOS 登记中断程序入口并进行硬件自检，再引导 OS。
- 存储位置：
  - BIOS/自举程序在 ROM。
  - OS 最终加载到 RAM 的系统区运行。
  - 活动分区的引导扇区中有用于引导 OS 的启动管理器。
- OS 引导不是把整个硬盘中的 OS 一次性全部加载到内存，常驻内存的主要是内核和初始化所需部分。

### 1.6 虚拟机

#### 1.6.1 虚拟机的基本概念

- 虚拟机：利用虚拟化技术把一台物理机器虚拟为多台虚拟机器，隐藏具体硬件差异，提供抽象、统一、模拟的计算环境。
- 第一类虚拟机管理程序 `Type 1`：
  - 直接运行在硬件上，处于最高特权级。
  - 又称裸金属架构。
  - 可在裸机上提供多个虚拟机，每个虚拟机可运行不同客户 OS。
  - 通常效率高于第二类。
- 第二类虚拟机管理程序 `Type 2`：
  - 依赖宿主 OS，像普通应用程序一样运行。
  - 又称寄居架构。
  - 例：VMware Workstation。
  - 底层 OS 为宿主操作系统；虚拟机中运行的是客户操作系统。
- 虚拟化状态：
  - 虚拟机作为用户态进程运行，不允许直接执行敏感指令。
  - 客户 OS 认为自己在内核态，实际处于虚拟内核态。
  - 客户 OS 执行敏感指令时陷入 VMM，由 VMM 检查并模拟执行。
- 多台虚拟机可同时运行在同一物理机上，但是否真正并行取决于物理处理器数量。

#### 1.6.2/1.6.3 本节习题精选与答案解析吸收

- 分层式 OS 结构清晰、便于调试；但依赖关系固定，不灵活，效率较低。
- 模块化 OS 的功能模块在内核中，模块出错可能导致整个内核崩溃；“模块错不影响整个系统”是微内核优势。
- 大内核/宏内核的缺点不是“内核切换太慢”；切换开销大是微内核问题。
- “功能分层设计”不是微内核四特征之一。
- 微内核中不宜放文件系统服务；文件系统通常在用户态服务器中。
- 微内核不是更高效；它更可靠、安全、可移植、可扩展。
- 外核负责资源分配、回收、保护和安全检查，不负责进程调度。
- 操作系统最终加载到 RAM，不是 BIOS、ROM、EPROM。
- OS 初始化/引导过程中需要创建或初始化中断向量表；硬盘分区表、文件系统根目录、索引节点表属于磁盘分区/格式化和文件系统创建阶段。
- 虚拟机可由软件或硬件实现；多台虚拟机同时运行不等于真正并行。
- VMware Workstation 是第二类 VMM，本质上是宿主 OS 上的应用程序；一台虚拟机可安装多个 OS。
- 第一类 VMM 直接运行在硬件上，通常效率高于第二类；VMM 代码量通常小于完整 OS；第二类 VMM 通常不运行在最高特权级。

### 1.7 本章疑难点

- 并行性与并发性：
  - 并行：同一时刻发生。
  - 并发：同一时间间隔内发生。
  - 单处理器系统中，多个程序只能交替并发；多处理器可把可并发程序分配到不同处理器上并行执行。
- 特权指令与非特权指令：
  - 特权指令权限大，必须在内核态执行。
  - 用户态使用特权指令会产生异常/中断，阻止用户破坏系统。
  - 从用户态进入内核态的唯一途径是中断或异常。
- 访管指令与访管中断：
  - 访管指令 `trap` 在用户态执行，本身不是特权指令。
  - 它有意识地产生自陷/访管中断，使 CPU 转入内核态，请求系统服务。
- 微内核 OS 四要点：
  - 内核足够小。
  - 基于客户/服务器模式。
  - 机制与策略分离。
  - 采用面向对象技术。

## 易错点/易混点

- OS 管理文件，但不关心源程序文件的具体内容；编译器不是 OS 必须提供的功能。
- GUI 不是 OS 的必要组成部分；GUI 调用的系统调用属于 OS。
- 并发是同一时间间隔，并行是同一时刻；单 CPU 上进程不并行。
- 并发和共享是最基本特征，虚拟和异步建立在多道程序环境之上。
- 多道程序设计不保证进程越多 CPU 利用率越高。
- 批处理系统缺点是无交互能力，不是吞吐量小。
- 分时系统追求快速响应，实时系统追求及时性和可靠性。
- 实时系统允许牺牲资源利用率来保证高优先级任务及时处理。
- 用户态不能执行 I/O、关中断、设置时钟、设置屏蔽字等特权指令。
- `trap` 指令可以在用户态执行；它的目标就是主动进入内核态。
- “系统调用发生在用户态”指用户态发起和执行陷入指令；具体服务例程在内核态执行。
- 进程切换只能发生在内核态；系统调用、缺页、外部中断可以在用户态发生。
- 中断/异常发生时 CPU 不一定处于内核态；中断处理程序开始执行时一定处于内核态。
- 缺页是异常/内中断，不是外部中断。
- 中断保存断点和普通子程序调用都保存 PC，但中断还必须保护 PSW。
- 系统调用接口因 OS 不同而不同，不存在所有 OS 统一的系统调用接口。
- 微内核优点不是性能高；性能开销大才是它的主要问题。
- 模块化不等于微内核；模块化内核中的模块错误可能导致整个内核崩溃。
- 外核不是完整 OS 内核，它不负责进程调度。
- 引导程序不会把整个 OS 一次性全部加载到内存。
- 多台虚拟机同时运行不等于真正并行，真正并行取决于物理处理器资源。

## 注解

- OS 定义背三词：管理资源、组织调度、提供接口。
- 四特征理解为：并发和共享是根，虚拟是手段/效果，异步是多道环境中的表现。
- 系统类型判断先看目标：吞吐量是批处理，交互响应是分时，规定时间处理外部事件是实时。
- 态切换判断：系统调用靠自陷，硬件事件靠外中断，程序错误靠异常。
- 特权指令判断：能直接影响系统安全、硬件状态、全局状态的，基本就是特权指令。
- 系统调用流程记作：`参数 -> trap -> 内核服务 -> 返回用户态`。
- 中断/系统调用分工记作：硬件存 PC/PSW 和改模式，OS 存通用寄存器并执行服务。
- 内核结构题抓关键词：宏内核“快但大”，微内核“小而稳但慢”。
- 引导流程题不要跳步：BIOS 找启动设备，MBR 找活动分区，PBR 找启动管理器。

## 速背检查

1. OS 是什么？管理计算机软硬件资源、组织调度工作并提供接口与环境的基本系统软件。
2. OS 管理哪四类资源？处理机、存储器、文件、设备。
3. OS 提供哪两类主要接口？命令接口和程序接口。
4. 程序接口由什么组成？系统调用。
5. OS 四个基本特征是什么？并发、共享、虚拟、异步。
6. 最基本的两个特征是什么？并发和共享。
7. 并发和并行区别？并发是同一时间间隔，并行是同一时刻。
8. 单道批处理特点？自动性、顺序性、单道性。
9. 多道批处理特点？多道、宏观并行、微观串行。
10. 分时系统特点？同时性、交互性、独立性、及时性。
11. 实时系统最重要目标？及时性和可靠性。
12. 用户态能执行特权指令吗？不能。
13. 从用户态进入内核态的唯一途径？中断或异常。
14. 访管指令在哪个态执行？用户态。
15. 系统调用服务例程在哪个态执行？内核态。
16. 外中断和异常区别？外中断来自 CPU 执行指令之外，异常来自指令执行内部。
17. 缺页属于什么？异常/内中断中的故障。
18. 时钟中断属于什么？外部中断。
19. 系统调用处理顺序？传参 -> trap -> 服务例程 -> 返回用户态。
20. 系统调用中硬件负责什么？保存 PC/PSW、切换到内核态。
21. 系统调用中 OS 负责什么？保存通用寄存器、执行服务例程、恢复现场。
22. 宏内核特点？功能集中在内核态，性能高但内核庞大。
23. 微内核主要问题？用户态/内核态频繁切换导致开销大。
24. 外核负责什么？分配、回收、保护资源并检查安全性。
25. OS 引导链条？`BIOS -> MBR -> PBR -> 启动管理器 -> OS`。
26. OS 最终加载到哪里？RAM。
27. 第一类 VMM 运行在哪里？直接运行在硬件上。
28. 第二类 VMM 依赖什么？宿主操作系统。
