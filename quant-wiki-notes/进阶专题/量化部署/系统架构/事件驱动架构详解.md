# 事件驱动架构详解

> [!note] 概述
> 事件驱动架构（Event-Driven Architecture）是量化交易系统的核心设计模式，使回测代码能无缝迁移到实盘环境。

## 什么是事件驱动？

在事件驱动系统中，所有操作都由**事件**触发，按时间顺序逐一处理：

```
时间线：09:30:00 → 09:30:01 → 09:30:02 → ...

事件流：
  MarketEvent (新K线/新Tick)
      ↓
  SignalEvent (策略产生信号)
      ↓
  OrderEvent (订单发送)
      ↓
  FillEvent (成交回报)
```

## 核心组件

### 1. 事件队列（Event Queue）

```python
from collections import deque

class EventQueue:
    """FIFO事件队列"""
    def __init__(self):
        self.queue = deque()

    def push(self, event):
        self.queue.append(event)

    def pop(self):
        return self.queue.popleft() if self.queue else None
```

## 回测 vs 实盘统一架构

```
        回测                        实盘
        ────                        ────
DataHandler(读CSV/DB)  ←→  DataHandler(读WebSocket/API)
Portfolio(模拟撮合)     ←→  Portfolio(真实成交)
Execution(模拟)         ←→  Execution(券商API)
```

**同源代码收益**：回测和实盘使用相同的Strategy、Portfolio、RiskManager代码，只替换DataHandler和ExecutionHandler的实现。

## 事件类型设计

```python
class EventType(Enum):
    MARKET = 'MARKET'         # 行情更新
    SIGNAL = 'SIGNAL'         # 交易信号
    ORDER = 'ORDER'           # 订单
    FILL = 'FILL'             # 成交
    RISK = 'RISK'             # 风控事件
    SYSTEM = 'SYSTEM'         # 系统事件（启动/关闭）
```

## Homalos 事件驱动系统

[Homalos](https://homalos.github.io/) 是一个开源的事件驱动量化系统：

- 数据中心：统一管理多数据源
- 事件总线：组件间异步解耦通信
- 策略容器：热加载/热更新策略
- 实时风控：规则引擎 + 动态阈值

## 设计要点

1. **单一职责**：每个组件只做一件事
2. **松耦合**：组件通过事件通信，不直接依赖
3. **可替换性**：DataHandler/ExecutionHandler可插拔
4. **时间一致性**：回测和实盘遵循相同的时间语义
5. **状态管理**：系统状态可序列化、可恢复

## 实战掌握清单

> [!tip] 交易者视角
> 事件驱动架构详解 的学习重点不是记住术语，而是把它放进研究、组合、执行和复盘的闭环。量化部署关注系统能否稳定、可追踪、可停止地运行，核心是数据、信号、订单、风控和监控链路。

### 关键判断

- 明确研究、回测、模拟、实盘之间的数据和代码一致性。
- 检查订单状态、成交回报、撤单、风控拒单和异常告警。
- 评估延迟、可用性、容灾和人工接管流程。

### 落地动作

1. 上线前做权限隔离、资金上限、品种白名单和熔断规则。
2. 保留行情、信号、订单、成交和持仓日志。
3. 定期演练断网、接口失败、数据异常和策略失控。

### 失效边界

- 没有 kill switch。
- 日志不完整。
- 研究环境与实盘环境口径不一致。

### 复盘问题

- 这项知识改变了哪一个具体决策：标的、方向、仓位、退出、对冲还是不交易？
- 如果判断相反，最大亏损、最长恢复期和退出触发条件是什么？
- 有没有一个更简单的基准方法可以取得相近结果？
