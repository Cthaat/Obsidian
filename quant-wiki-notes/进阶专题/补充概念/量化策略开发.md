# 量化策略开发

> [!note] 核心定位
> 量化策略开发 属于策略研究和组合落地主题。重点是先定义收益来源，再验证数据、成本、风险、容量和失效条件，最后写成可监控的交易流程。

## 一、核心定义

量化策略开发 是把投资逻辑转化为规则、数据、回测、组合和执行的过程。策略不是想法本身，而是可复现的收益来源、风险预算和操作纪律。

## 二、开发流程

完整流程包括：提出假设、定义数据、实现信号、做回测、扣成本、做样本外、评估容量、设计组合、上线监控和定期复盘。跳过任何一步都会把研究风险转成实盘亏损。

## 三、风险来源

策略可能死于过拟合、成本低估、容量不足、流动性枯竭、参数漂移、市场结构变化和执行异常。高收益策略如果无法解释风险来源，不应直接加仓。

## 四、最小策略文档

策略名称、收益来源、交易品种、数据口径、信号公式、调仓频率、成本假设、仓位规则、最大回撤、停止条件、上线监控、复盘频率。

## 六、策略是否值得进入实盘

`量化策略开发` 必须通过“研究有效”和“实盘可执行”两道门。很多策略回测赚钱，但一加入成本、滑点、容量、风控和心理承受力，就不再适合真实账户。

| 检查项 | 最低要求 |
|---|---|
| 收益来源 | 能解释，不依赖玄学参数 |
| 数据 | 交易时点真实可得 |
| 回测 | 无前视、无幸存者偏差 |
| 成本 | 扣除佣金、税费、滑点、冲击成本 |
| 风险 | 最大回撤和最长恢复期可承受 |
| 容量 | 资金规模扩大后仍可成交 |
| 运维 | 有监控、告警和暂停机制 |

策略开发不是为了找到一次性圣杯，而是建立不断验证、淘汰、迭代和控制风险的流程。

## 八、策略上线前的最后一问

`量化策略开发` 上线前，要从“能赚钱”转向“亏钱时怎么办”。策略在历史上赚钱并不稀奇，能解释亏损、限制亏损、复盘亏损，才是真正可用。

| 问题 | 最低答案 |
|---|---|
| 为什么赚钱 | 清晰收益来源，不依赖玄学参数 |
| 为什么会亏 | 明确失效环境和风险来源 |
| 亏多少停 | 有回撤、波动、连续亏损阈值 |
| 如何降级 | 降仓、暂停、回到基准策略 |
| 如何复盘 | 数据、信号、执行、组合、心理分开记录 |

## 深度案例与训练

### 策略拆解

学习 量化策略开发 时，写清楚策略的六个零件：信号、过滤器、入场、退出、仓位和风控。任何说不清这六项的策略，都还只是想法。

### 验证路径

1. 用最简单规则建立基准收益。
2. 加入手续费、滑点、冲击成本和容量约束。
3. 做样本外、滚动窗口和参数扰动。
4. 统计收益来自哪些时期、哪些品种和哪些市场状态。

### 实盘监控

上线后关注信号命中率、成交质量、回撤、换手、资金占用和异常订单。策略失效时优先降仓或暂停，而不是继续调参寻找安慰。

## 最小可执行项目

### 策略一页纸

围绕 量化策略开发 写一张策略一页纸：收益来源、适用品种、信号、过滤器、入场、退出、仓位、风控、成本、容量、暂停条件和复盘指标。

| 检验 | 合格标准 |
|---|---|
| 逻辑 | 能解释为什么会有收益 |
| 回测 | 有样本外和参数扰动 |
| 成本 | 扣除手续费、滑点和冲击成本 |
| 风险 | 有最大回撤和连续亏损预案 |
| 实盘 | 有监控、降仓和暂停规则 |

### 常见失败

策略在报告中赚钱，却在实盘中输给交易成本、容量限制、信号衰减和执行错误。


## 进一步掌握

学习 `量化策略开发` 时，要把它当作研究基础设施，而不是单独的工具说明。真正重要的是数据能否复现、字段口径是否稳定、异常能否被发现，以及研究结果能否安全地走到实盘或组合复盘。一个工具只要影响数据、信号、订单或日志，就已经进入风险管理范围。

| 检查维度 | 应用方法 | 常见错误 |
|---|---|---|
| 数据契约 | 记录字段、频率、复权、时区、更新时间 | 只看能否下载，不看口径变化 |
| 复现能力 | 保存参数、版本、原始数据和处理脚本 | 回测结果无法重跑 |
| 生产边界 | 区分研究、仿真、小资金和正式账户 | 研究脚本误触实盘权限 |

### 训练任务

围绕 `量化策略开发` 做一次小型验收：选一个真实标的或数据表，记录获取路径、字段解释、缺失值、异常值、更新频率和与第二数据源的差异。最后写出“什么时候必须停止使用这份数据”的条件。掌握工具的标志不是会调用接口，而是能解释数据错误会怎样传导成投资错误。
