# AI高频交易前沿

> [!note] 概述
> 高频交易（HFT）是机器学习最具挑战性的应用领域之一。微秒级的决策要求、海量数据流、极低信噪比，使得传统ML方法需要特殊适配。

## HFT对ML的特殊要求

| 要求 | 传统ML | HFT ML |
|------|--------|--------|
| 延迟 | 毫秒-秒级 | 微秒级 |
| 数据量 | GB级/天 | TB级/天 |
| 特征工程 | 离线预计算 | 在线流式计算 |
| 模型推理 | CPU/GPU | FPGA/定制硬件 |
| 更新频率 | 周/月 | 实时在线 |

## HFT中的核心ML应用

### 1. 订单流预测

```
订单簿（LOB）数据：
├── 买卖挂单量（5档/10档）
├── 成交记录（逐笔）
├── 撤单信息
└── 价差变化

→ LSTM/CNN → 预测短期价格方向（毫秒-秒级）
```

### 2. 做市商策略优化

- 预测最优买卖报价
- 库存风险管理
- 对手方行为建模

### 3. 异常检测

- 市场操纵识别（刷单、幌骗）
- 流动性枯竭预警
- 闪崩风险预测

### 4. 执行算法

- VWAP/TWAP执行的ML增强
- 智能订单路由
- 流动性预测

| 层级 | 需求 | 实现方案 |
|------|------|---------|

| 存储 | 时序DB | ClickHouse/Kdb+ |

## 关键挑战

1. **信噪比极低**：Tick级别数据99%是噪声
2. **分布漂移快**：市场微观结构持续变化
3. **竞争对手也在用ML**：不断军备竞赛
4. **成本极高**：机房、数据、硬件投入百万起
5. **监管风险**：交易所/监管对HFT审查趋严

## 个人/HFT入门建议

1. 从比特币交易所开始（门槛低、数据公开）
2. 使用L2（订单簿）数据而非Tick
3. 从分钟/秒级做起，再逐步到毫秒级
4. 关注做市策略（稳定性优于投机性策略）
5. 成本建模比信号生成更重要

## 课程化学习补充

> [!important] 学习定位
> 机器学习在交易中是预测和排序工具，不是收益机器；关键在数据切分、特征稳定性、可解释性和上线监控。本文仅用于学习、研究与复盘，不构成任何投资建议。

### 必须掌握的问题

- 训练/验证/测试是否按时间切分
- 特征是否可在交易时点获得
- 收益标签是否扣成本
- 模型漂移是否有监控

### 实战应用流程

1. 先写清楚你的投资假设：为什么这个信号、资产或方法应该产生收益。
2. 明确数据口径：样本范围、更新时间、复权/分红/停牌处理和交易日历。
3. 做最小可行验证：先用简单规则验证方向，再逐步加入复杂模型。
4. 把成本和约束前置：手续费、滑点、冲击成本、保证金、流动性和容量都要进入测算。
5. 上线后持续复盘：记录信号、下单、成交、持仓、回撤和失效原因。

### 风险与失效条件

- 随机切分造成泄漏
- 高维特征过拟合
- 黑箱模型难以风控
- 训练收益无法覆盖交易成本

### 复盘问题

- 这笔交易或这套模型赚的是什么钱：风险补偿、行为偏差、流动性溢价，还是偶然噪音？
- 如果市场环境反过来，最大亏损和最长恢复期会是多少？
- 当前结论是否依赖某个不可持续假设，例如低利率、低波动、充裕流动性或监管套利？
- 有没有一个更简单的基准策略能取得接近效果？

## 跨领域进阶扩展

> [!tip] 交易者视角
> 学到 `AI高频交易前沿` 时，不要只把它当成孤立知识点。把机器学习当成预测、排序和特征压缩工具，重点是防泄漏、可解释和上线监控。优秀投资交易者会把它放入“宏观背景 - 资产选择 - 估值/信号 - 组合风险 - 交易执行 - 复盘反馈”的闭环。

### 与其他知识的连接

- 时间序列切分和标签设计
- 特征稳定性与模型漂移
- 交易成本后的目标函数
- 风控约束和模型治理

### 进阶训练

1. 检查每个特征是否在交易时点可得
2. 用时间滚动验证替代随机切分
3. 给模型建立漂移、胜率和换手监控

### 能力验收

- 能否说清楚这个主题影响的是收益来源、风险来源、交易成本、流动性还是心理纪律？
- 能否指出它在什么市场环境、资产类别或交易周期中更有效？
- 能否把它写成一条可复盘的研究或交易规则？
- 能否说明如果判断错误，组合最大损失和退出机制是什么？
